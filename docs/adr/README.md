# Architecture Decision Records (ADR)

ADRs capture **important architectural decisions** with context and trade-offs.

## When to write an ADR

Write an ADR when you make a change that affects:

- architecture or major dependencies
- deployment topology
- core data flow
- security or secrets handling

## Template

Use this structure:

- **Title**
- **Status** (proposed / accepted / deprecated)
- **Context** (why this decision matters)
- **Decision** (what we chose)
- **Consequences** (trade-offs, follow-ups)

## Naming

- `NNNN-short-title.md` (e.g. `0002-add-redis-cache.md`)
